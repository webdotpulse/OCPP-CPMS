# Energy Management System (EMS) Gateway Manual

This technical manual details how external Energy Management Systems (EMS) can integrate with the Open-Source OCPP CMS. The system provides a dedicated `EmsGatewayService` for ingesting real-time building load, solar generation, and battery status to influence the dynamic smart charging algorithm.

---

## 1. Overview

The CMS allows you to map specific chargers (or groups of chargers) to an **EMS Gateway**. This gateway acts as an integration point for local energy controllers (e.g., a home energy manager or commercial building controller).

When an EMS pushes telemetry data (Grid Power, Solar Power, Battery Power), the CMS caches this data in **Redis** and feeds it into the `LoadManagementService`. This allows the CMS to dynamically adjust the charging speed of connected vehicles in real-time, preventing grid overloads and maximizing self-consumption of solar energy.

---

## 2. Authentication

Unlike standard user API routes which use JWTs, the EMS Gateway routes use a static, hardware-based **Authentication Token** (`auth_token`).

1.  **Create an EMS Gateway:** In the CMS Dashboard, navigate to **Settings** -> **EMS Gateways** -> **Add Gateway**.
2.  **Generate Token:** The system will generate a unique `auth_token` for this specific gateway.
3.  **Use Token:** The external EMS must include this token in the `Authorization` header of all HTTP requests.

```http
Authorization: Bearer <your_auth_token>
```

## 3. Telemetry Integration

The EMS should push real-time telemetry updates to the CMS at a regular interval (e.g., every 5 to 15 seconds).

### 3.1 Endpoint
`POST /api/ems-gateways/:id/telemetry`

*   `:id` is the UUID of the EMS Gateway.

### 3.2 Payload Schema
All values should be in Watts (W). Positive values indicate consumption/import, while negative values indicate generation/export.

```json
{
  "gridImportPower": 4500,    // Power currently being drawn from the grid
  "solarPower": -2000,        // Power currently being generated by solar (negative)
  "batteryPower": 0,          // Battery charge (positive) or discharge (negative)
  "houseLoad": 1500           // Uncontrolled building load (excluding EV chargers)
}
```

### 3.3 Redis Caching Mechanism
To ensure ultra-fast read performance for the Load Management Service and real-time dashboard updates, the telemetry data is not immediately written to the PostgreSQL database.

Instead, the `EmsGatewayService` caches the payload in a **Redis Hash**:
*   **Key:** `ems_telemetry:<gateway_id>`
*   **TTL:** The cache key automatically expires after 300 seconds (5 minutes) if no new data is received, acting as a dead-man's switch.

A separate background cron job (`EmsTelemetrySyncService`) periodically synchronizes the latest Redis state to the PostgreSQL `EmsTelemetryRecord` table for historical reporting.

## 4. Load Management and Remote Control

The CMS `LoadManagementService` evaluates active charging sessions against the configured `maxPower` limits of the Station or Charge Group, taking into account the real-time `gridImportPower` provided by the EMS.

### 4.1 Internal Algorithm
If the theoretical max load of all active charging sessions exceeds the safe limit, the algorithm triggers a load-balancing event.
The system calculates a safe `limitW` or `limitAmps` per transaction and dispatches an OCPP `SetChargingProfile` command to the physical charger.

### 4.2 Direct Control Endpoint
External EMS systems can bypass the internal Load Management algorithm and send direct `SetChargingProfile` commands to specific chargers via the gateway.

**Endpoint:** `POST /api/ems-gateways/:id/remote-control/set-profile`

**Payload:**
```json
{
  "chargerId": 1,
  "connectorId": 1,
  "csChargingProfiles": {
    "chargingProfileId": 100,
    "stackLevel": 1,
    "chargingProfilePurpose": "ChargePointMaxProfile",
    "chargingProfileKind": "Absolute",
    "chargingSchedule": {
      "chargingRateUnit": "A",
      "chargingSchedulePeriod": [
        {
          "startPeriod": 0,
          "limit": 16
        }
      ]
    }
  }
}
```
*Note: `chargerId` must be a numeric integer mapping to the Prisma `Charger.charger_id`.*
